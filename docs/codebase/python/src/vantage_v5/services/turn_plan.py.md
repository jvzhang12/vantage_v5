# `src/vantage_v5/services/turn_plan.py`

Internal TurnPlan DTOs for Vantage chat turns.

## Purpose

- Build a typed, trace-safe `TurnPlan` from already-finalized `/api/chat` request and response fields.
- Capture retrieval selection, explicit UI surface intent, write/draft intent, a trace-backed write ledger, a write-intent projection, visible context, side-effect policy, semantic/protocol summaries, execution policy, compatibility links, and trace-only validation warnings.
- Provide authoritative TurnPlan helpers for UI surface actions, no-write suppression, and narrow write candidate gates. Open/close/preserve surface application consumes `TurnPlanSurfaceAuthority`; Whiteboard draft/offer materialization consumes `TurnPlanDraftAuthority`; and the other positive write gates validate only existing structured candidates. Routing, retrieval, frontend behavior, and draft/content generation still use their existing paths.

## Key Classes / Functions

- `TurnPlan`: top-level frozen dataclass with `request`, `route`, `retrieval`, `visible_context`, `ui_surface_action`, `write_intent`, `side_effect_policy`, `write_ledger`, `protocols`, `semantic`, `execution`, `compatibility`, and `validation` sections.
- `TurnPlanSurfaceAuthority`: narrowed execution-facing view derived from TurnPlan surface, write, and side-effect fields. It centralizes whether a surface turn is close, preserve, or Whiteboard open-only; whether writes are forbidden; the no-write reason/category; whether auto graph writes, protocol writes, local semantic save/publish actions, and artifact actions must be suppressed; and whether operational surface payloads should be built.
- `TurnPlanExecutionPolicy`: small internal handoff object that bundles TurnPlan surface, artifact-write, and draft/offer authority verdicts for the orchestrator and `ChatService.reply()` call. It keeps the chat-path suppression kwargs, denied-draft receipt checks, local semantic write gates, and hard no-write reason handling on one TurnPlan-derived policy object.
- `TurnPlanArtifactWriteAuthority`: narrowed execution-facing permission gate for local artifact save/publish candidates. It permits `artifact_save` / `artifact_publish` only when structured semantic/control-panel artifact write intent exists, hard no-write authority is absent, and a safe current artifact target is available.
- `TurnPlanMemoryWriteAuthority`: narrowed execution-facing permission gate for memory write candidates. It permits `memory_write` only when structured remember/memory intent exists, hard no-write authority is absent, and safe memory content is available.
- `TurnPlanConceptWriteAuthority`: narrowed execution-facing permission gate for concept write candidates. It permits concept writes only when structured control-panel or semantic concept intent is present alongside an existing concept candidate, hard no-write authority is absent, persisted concept content is available, and revision targets are safe.
- `TurnPlanProtocolWriteAuthority`: narrowed execution-facing permission gate for protocol upsert/update candidates. It permits protocol writes only when an existing structured protocol-interpreter/control/semantic candidate is present, hard no-write authority is absent, persisted protocol fields are available, the target protocol id/kind is safe, and the existing protocol write policy would have allowed the update.
- `TurnPlanOperationalProposalAuthority`: narrowed execution-facing permission gate for calendar/task proposal candidates. It permits proposed operational actions only when existing compiler/control-plane structured candidates are present, hard no-write authority is absent, action details and targets are safe, every candidate has exact `status="proposed"`, and every candidate is confirmation-gated.
- `TurnPlanDraftAuthority`: narrowed execution-facing permission gate for pending Whiteboard draft/offer candidates. It permits `whiteboard_draft` or `whiteboard_offer` only when structured request, turn-interpretation, control-panel, or surface-invocation authority exists and hard no-write surface authority is absent; if no-op preserve stripped a structured draft control, the suppressed control-panel metadata can still be used as a denial source so receipts stay truthful without reopening raw-text routing.
- `WriteLedgerPlan` / `WriteLedgerEntry`: compact trace-only ledger of finalized write effects, including no-write, open-only/no-write, pending Whiteboard offers/drafts, workspace updates, artifact saves/promotions, protocol writes, concept writes, memory writes, and proposed or accepted calendar/task mutations.
- `WriteProjectionPlan`: trace/debug projection that aligns structured write-intent sources such as semantic policy, control-panel actions, memory intent, protocol interpreter effects, and finalized write effects with the write ledger.
- `TurnPlanValidation`: trace-only warning container used to compare finalized TurnPlan sections against the final public response fields.
- `TurnPlanBuilder`: builds a `TurnPlan` from a final trace request payload plus the final public response payload.
- `build_turn_plan_surface_authority()`: builds the internal surface-action authority from finalized or in-flight response fields, including nested `surface_invocation.surface_action` compatibility payloads.
- `is_turn_plan_hard_no_write_reason()`: shared helper for hard no-write reason checks, including a strict mode for paths where soft artifact-Q&A suppression must not count as a surface-level hard boundary.
- `turn_plan_trace_payload()`: convenience helper used by final-response trace persistence.
- `project_write_intent_compatibility()`: post-execution compatibility annotator that adds explicit write-intent/effect metadata to `surface_invocation` and upgrades generic chat/no-write-looking fields only when finalized structured write intent or committed write effects already exist.

## Notable Behavior

- Selected attention resources are compacted for TurnPlan traces; full resource content is not duplicated inside the plan.
- Whiteboard `open_only` is recorded as a UI-only action and a no-write side-effect policy. The narrowed surface-authority helper is consumed by the orchestrator and server post-processing to keep this path chat/no-write.
- Final `surface_action` close directives are recorded as UI surface actions with a no-write side-effect policy. The narrowed surface-authority helper is consumed to apply close acknowledgements and suppress downstream write/action compilation.
- `preserve_visible_surface` invocations are recorded as no-op UI preservation with a no-write side-effect policy and no operational surface payloads.
- Visible/selected artifact Q&A is recorded as chat-first/no-write policy when the existing structured `surface_invocation.intent` already says the turn was a current-artifact or selected-material follow-up, the selected/visible context is an openable artifact or Whiteboard item, and no structured write authority is present. Structured write authority includes request-level `memory_intent="remember"`, control-plane or semantic save/learn/remember fields, real semantic policy actions such as `artifact_save` and `artifact_publish`, and protocol upsert effects already produced by the protocol interpreter. Operational surfaces and ordinary non-artifact Q&A continue through the existing meta path.
- Hard no-write surface authority (`open_only`, close, or preserve) is also reflected in execution policy as disabled local semantic writes, so local `artifact_save` / `artifact_publish` handlers cannot leak write payloads into UI-only or no-op surface turns.
- Artifact save/publish positive authority is narrow and structured: `TurnPlanArtifactWriteAuthority` can allow existing local semantic save/publish candidates through, or deny them with a trace reason such as `open_only_ui_handoff`, `close_visible_surface`, `preserve_visible_surface`, `missing_structured_artifact_write_intent`, or `artifact_write_target_unavailable_or_ambiguous`. It does not infer save/publish intent from raw user text and does not create write actions.
- Memory positive authority is narrow and structured: `TurnPlanMemoryWriteAuthority` can allow existing `create_memory` candidates through when request-level `memory_intent="remember"` or structured control/semantic memory intent exists and the candidate exposes non-empty persisted memory fields (`title`, `card`, or `body`), or deny them with a trace reason such as `open_only_ui_handoff`, `close_visible_surface`, `preserve_visible_surface`, `missing_structured_memory_write_intent`, or `memory_write_content_unavailable_or_unsafe`. It does not infer remember intent or safe memory content from raw user text, assistant text, visible artifacts, or workspace content, and it does not create memory actions.
- Concept positive authority is narrow and structured: `TurnPlanConceptWriteAuthority` can allow existing `create_concept` / `create_revision` candidates through only when structured control-panel or semantic concept intent is also present, the candidate exposes non-empty persisted concept fields (`title`, `card`, or `body`), hard no-write authority is absent, and revisions name a target concept. A meta candidate by itself is treated as a candidate, not semantic authority, so ordinary Q&A cannot become a durable concept without explicit learn/conceptualize/create-concept intent. It can deny candidates with reasons such as `open_only_ui_handoff`, `close_visible_surface`, `preserve_visible_surface`, `missing_structured_concept_write_intent`, `concept_write_content_unavailable_or_unsafe`, or `concept_write_target_unavailable_or_ambiguous`. It does not infer concept intent or safe concept content from raw user, assistant, visible artifact, or workspace text, and it does not create concept actions.
- `write_ledger` is built only from finalized response payload fields such as `workspace_update`, `graph_action`, `created_record`, `artifact_actions`, `surface_invocation.write_behavior`, and `surface_action`; it does not infer semantic write intent from raw user text and does not affect execution.
- Protocol positive authority is narrow and structured: `TurnPlanProtocolWriteAuthority` can allow existing `upsert_protocol` candidates from the protocol interpreter through when the candidate exposes safe persisted protocol fields (`title`, `card`, or `body`), a concrete protocol id/kind target, no hard no-write authority, and the pre-existing reusable-preference/protocol-edit policy also allows the write. It can deny candidates with reasons such as `open_only_ui_handoff`, `close_visible_surface`, `preserve_visible_surface`, `protocol_write_blocked_by_existing_policy`, `missing_structured_protocol_write_intent`, `protocol_write_content_unavailable_or_unsafe`, or `protocol_write_target_unavailable_or_ambiguous`. It does not infer protocol intent from raw user text and does not create protocol actions.
- Operational proposal authority is narrow and structured: `TurnPlanOperationalProposalAuthority` can allow existing calendar/task proposal candidates from the artifact mutation compiler through only when action content and targets are available, each candidate has exact `status="proposed"`, all candidates require confirmation, hard no-write authority is absent, and the candidate remains a proposal rather than an accepted mutation. It can deny candidates with reasons such as `open_only_ui_handoff`, `close_visible_surface`, `preserve_visible_surface`, `missing_structured_operational_mutation_intent`, `operational_proposal_content_unavailable_or_unsafe`, `operational_proposal_target_unavailable_or_ambiguous`, `operational_proposal_requires_proposed_status`, or `operational_proposal_requires_confirmation`. It does not infer operational mutation intent from raw user text and does not execute accepted calendar/task mutations.
- Whiteboard draft/offer authority is narrow and structured: `TurnPlanDraftAuthority` can allow existing pending Whiteboard draft or offer candidates through only when explicit composer mode, turn interpretation, Navigator/control-panel, or surface-invocation fields already carry draft/offer authority and no hard no-write reason is active. It can also recognize a `control_panel.suppressed_actions` draft entry as a denied candidate source when preserve/close normalization already removed that draft from active controls. It can deny candidates with reasons such as `open_only_ui_handoff`, `close_visible_surface`, `preserve_visible_surface`, `artifact_qna_chat_first`, or `missing_structured_whiteboard_draft_intent`. It does not infer draft intent from raw user text and does not generate draft content.
- `write_projection` records intended write kinds, authority/source, actual write categories, and intent/effect agreement from structured fields only. It recognizes semantic policy `artifact_save` / `artifact_publish`, control-panel write actions, `memory_intent="remember"`, structured concept authority, protocol write candidates/authority/effects, existing write-intent plan fields, and finalized write effects.
- Compatibility projection is additive and post-execution: explicit save/publish/memory/protocol writes can annotate `surface_invocation.write_intent` and `surface_invocation.write_effects`, preserving legacy fields where useful while avoiding final traces that look like generic no-write chat. Protocol upserts that are stored through the concept store still project as `protocol_write`, can override legacy memory/concept-looking surface intents and nested surface statuses in the final compatibility payload, and remove memory/concept write kinds from the projection unless a separate real memory/concept write effect also occurred.
- TurnPlan is authoritative for surface action application, for suppressing writes on structured no-write turns, for permitting pending Whiteboard draft/offer candidates that already have structured draft/offer intent, for permitting local artifact save/publish candidates that already have structured write intent, for permitting memory write candidates that already have structured memory intent, for permitting concept write candidates that already have structured concept intent plus a safe candidate, for permitting protocol update candidates that already exist in the protocol interpreter path, and for permitting calendar/task proposal candidates that already exist in the artifact mutation compiler path. It still does not infer or create writes; Whiteboard draft generation and accepted calendar/task mutation execution remain on their existing paths.
- Validation warnings are non-authoritative diagnostics only. They flag contradictions such as selected context opening a UI surface without explicit open authority, `open_only` carrying writes, saved-artifact opens that lack a selected openable target, UI open targets that conflict with the selected primary resource, preserve/close turns being reclassified or carrying side effects, visible/selected-artifact Q&A writing without explicit write authority, draft surface mismatches, operational active-surface/payload mismatches, and calendar/task mutations that are not proposal-only/confirmation-gated.
- Public compatibility fields remain unchanged; TurnPlan records where existing fields such as `navigator_selection`, `surface_invocation`, `workspace_update`, `graph_action`, `created_record`, and `artifact_actions` appeared in the final response.
