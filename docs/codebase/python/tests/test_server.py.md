# `tests/test_server.py`

This test file covers the HTTP surface of the app and the end-to-end flows built on top of it. It verifies health and workspace endpoints, memory and concept listing, concept and vault-note search, individual record lookups, chat response payload shape, canonical `recall` payloads plus the legacy `working_memory` alias, response-mode payload fields, top-level `turn_interpretation` payloads for normal chat and Scenario Lab turns, the newer `workspace.context_scope` contract that distinguishes hidden whiteboards from in-scope ones, explicit top-level `selected_record` / `selected_record_id` disclosure for selected-item continuity, generic saved-item opening via `record_id`, branch-workspace reopening through `/api/workspace/open`, missing-workspace 404 behavior for that reopen route, graph-action payload compatibility, navigator-routed Scenario Lab turns that save branch workspaces plus a comparison artifact, stable `scenario_comparison` / `scenario_kind="comparison"` markers on that artifact, neutralized comparison headings that stay aligned with the saved artifact title, nested `scenario` metadata on reopened branch workspaces and saved comparison artifacts, detached branch namespaces for generic scenarios, legacy comparison artifacts whose base/branch lineage must still recover from `comes_from` when Scenario Lab serializes a selected record, follow-up turns that stay in normal chat when a scenario comparison artifact is selected, explicit Scenario Lab failure payloads that still return a chat answer, Scenario Lab turns that stay grounded when only whiteboard, recent-chat, or pending-whiteboard context is available, Scenario Lab turns that remain explicit best guesses when no grounded context exists, low-context follow-ups that keep an explicitly selected concept in the turn memory, longer semantic follow-ups that stay anchored because the navigator explicitly preserves the selected record, and off-topic follow-ups that do not leak a selected record back into working memory when preservation is not requested, including short off-topic turns against a selected scenario comparison artifact. It also covers first-class Memory Trace writes for ordinary chat turns, retrieval of recent Memory Trace records without promoting them into concepts or memories, experiment-scoped Memory Trace isolation, best-guess replies that are visibly prefaced with `This is new to me, but my best guess is:` when no grounded context supports them, work-product turns that can offer whiteboard collaboration, auto-mode whiteboard invitations driven by the navigator’s semantic hint, explicit user phrasing that bypasses the offer step and routes straight into whiteboard drafting, pending whiteboard offers that can carry into the next turn so a short acceptance can route into `draft`, thin alias-bridge coverage where pending whiteboard context backfills canonical `workspace_update.status` from the older `type` field, positive and negative coverage for the client-matched pending-carry phrase families (`let's do that`, `that works`, `resume that draft`, overlong follow-ups, and generic continue-without-reference turns), truthful `response_mode` disclosures for dedicated whiteboard acceptance and pending-offer follow-ups, live whiteboard follow-up edits that should update the active draft instead of reoffering a fresh whiteboard, signature/greeting-style edit follow-ups on visible drafts, defensive downgrades that prevent offer-mode turns from silently shipping a real whiteboard draft, dedicated whiteboard-accept requests that must replay the original work-product prompt instead of a synthetic acceptance message, whiteboard-drafting turns that now return canonical `workspace_update.status=\"draft_ready\"` proposals with `proposal_kind` metadata and no user-resolution decision yet, chat requests that use the current unsaved whiteboard buffer as transient context without silently persisting it, hidden unsaved whiteboards and even explicitly supplied live whiteboard buffers that should not accidentally ground ordinary chat when the scope is excluded, stale pending whiteboard offers that should be dropped for unrelated hidden-whiteboard chat turns, pending payloads without `origin_user_message` that should not count as carryable whiteboard context, explicit workspace saves that can target a freshly forked workspace id, save-plus-reopen coverage that proves an edited branch workspace keeps its durable Scenario Lab identity even when the submitted content omitted the metadata block, chat requests that can keep operating against an unsaved forked whiteboard buffer before that draft has been saved, explicit keep-in-chat whiteboard overrides, hard guards that block both implicit and same-turn explicit durable writes on whiteboard offer/draft turns, concept promotion, memory creation from chat intent, draft behavior that should not auto-write, transient workspace traces that should redact unsaved whiteboard content, unexpected chat failure handling, trace file uniqueness, experiment-mode isolation for temporary memory, and the helper paths that must preserve scenario metadata when a saved branch workspace is hidden or overlaid with a live buffer.

That matters because these routes are the contract for the whole application. The tests ensure the server exposes consistent data, routes requests to the right storage scope, avoids writing when it should only draft, keeps experiment data separate from durable repository state, exercises the navigator-first Scenario Lab path, confirms both conservative and interpreter-driven continuity anchoring, verifies grounded turns expose the exact grounded context surfaced for the final answer rather than only recalled working-memory hits, ensures best-guess turns are disclosed honestly in the visible assistant reply, locks in the newer behavior where the interpreter’s chosen path is visible through a compact `turn_interpretation` payload, confirms that concrete work products can first invite whiteboard collaboration and then return pending whiteboard proposals without being silently applied or promoted into durable memory, verifies that pending whiteboard context is reported only when the server had the original prior-turn prompt and a client-matched carry phrase to justify it, verifies the C3 alias bridge where pending whiteboard `type` can still recover the canonical `status`, verifies that stale pending offers are dropped when a hidden whiteboard should stay out of scope, verifies that even a supplied live whiteboard buffer stays out of model scope when the request explicitly excludes it, verifies that a direct user request for the whiteboard bypasses that invitation layer instead of redundantly asking again, verifies that visible-draft edit follow-ups keep using the current whiteboard even for signature- and greeting-level revisions, verifies that same-turn `remember` or `save` intent cannot punch through a pending whiteboard draft, checks that Scenario Lab comparisons carry stable metadata instead of depending on heading heuristics, verifies that neutralized comparison titles and H1 headings stay aligned, verifies that saved comparison artifacts and reopened branch workspaces expose durable scenario metadata through the normal server payloads, verifies that legacy comparison artifacts still recover durable lineage from `comes_from` when Scenario Lab carries a selected record forward, verifies that edited branch saves preserve branch identity across save and reopen, verifies that missing workspace-open requests now fail as clean 404s, verifies that Scenario Lab response modes stay truthful when they are grounded by whiteboard, recent chat, or pending whiteboard context, verifies that Scenario Lab fallback payloads stay explicit when nothing relevant grounds the turn, verifies that Scenario Lab failures stay explicit in the payload even when the server falls back to normal chat, and verifies that selected-record continuity is surfaced explicitly in the payload while also not being smuggled into the model on off-topic turns when it is absent from `working_memory`. The fixture also creates clean test `state/` and `traces/` directories so local manual experiments or trace files cannot leak into server assertions.
