# `src/vantage_v5/webapp/app.js`

Main browser entrypoint for the Vantage UI. It owns the client-side state tree, boots the app, talks to the FastAPI API, coordinates chat and whiteboard behavior, and renders the three major surfaces: chat, whiteboard, and `Vantage`.

## Purpose

- Bootstrap health, workspace, catalog, and persisted turn state.
- Manage the client-side state model for surfaces, Recall-facing turn state, Reasoning Path state, Memory Trace inspection state, whiteboard state, experiment state, and notices.
- Send chat requests with the correct workspace scope, pinned-context continuity, and pending whiteboard context.
- Render compact chat evidence, whiteboard decisions, guided-inspection summaries, a dedicated `Working Memory` narrative layer, the staged `Reasoning Path` rail, turn-scoped `Recall`, turn-scoped `Memory Trace`, inspectable learned items, a turn-scoped learned correction panel, Scenario Lab review, temporarily hidden library state, whiteboard lifecycle cues, and read-surface math rendering using the same backend-truth labels in both chat and Vantage.
- Prefer product-level recall explanations over raw backend vetting/debug strings when rendering Recall cards and turn notices.
- Keep the default shell calm: one transient global notice at a time, one secondary experiment toggle instead of a full experiment control row, and a shorter first-read Vantage summary.

## Key Responsibilities

- `boot()`, `loadHealth()`, `loadWorkspace()`, `refreshRuntimeStatus()`: startup and passive refresh flow. `restoreTurnSnapshot()` now marks a one-shot boot restore state that `loadWorkspace()` consumes so the restore/load contract is explicit rather than implicit.
- Startup now fails loud instead of silently: top-level window error handlers and the guarded `boot()` call push a UI notice/system message if frontend boot throws after the module begins executing.
- `sendMessage()`, `applyChatPayload()`, `absorbGraphNotices()`: chat request/response loop.
- `sendMessage()` now also handles a local reopen fast path for deictic follow-ups like `pull that up on the whiteboard` when the previous turn exposed exactly one reopenable recalled durable item.
- `sendMessage()` now handles experiment-mode questions and explicit end-experiment requests locally, so Vantage does not ask what experiment mode means after introducing it. Ending an experiment is confirmed in the browser before temporary notes are discarded.
- `applyWorkspacePayload()`, `applyWorkspaceDraft()`, `startFreshWorkspaceDraft()`, `saveWorkspace()`: whiteboard state transitions. `saveWorkspace()` now also consumes the returned artifact snapshot metadata so whiteboard saves leave a visible durable work product trail in the library, notices, and the whiteboard-owned latest-work-product cue.
- `acceptPendingWorkspaceOffer()`, `applyPendingWorkspaceUpdate()`: offer-first whiteboard collaboration flow.
- `renderWorkspacePreview()`: keeps the whiteboard source editor and the rendered preview distinct. The preview only appears when the draft includes renderable math or code, now through the shared `deriveWhiteboardPreviewState()` gate, uses the shared rich-text/math renderer, and leaves the textarea as the raw source of truth.
- `renderWorkspaceMeta()` and `renderWorkspaceArtifactCue()`: keep the whiteboard title deck, lifecycle text, and latest saved work-product cue aligned with the current draft state while preserving artifact lifecycle meaning such as `saved whiteboard` versus `promoted artifact`.
- `currentTurnGrounding()`: thin wrapper over the shared `deriveTurnGrounding()` helper from `product_identity.mjs`, so the app consumes one normalized grounding view-model instead of re-deriving counts and labels locally.
- `renderChatSurfaceCopy()`: owns the lighter Pass 02A chat-first shell copy. It now keeps the default masthead shorter, preserves a reduced experiment cue in whiteboard-sidebar mode, and adjusts the subtitle instead of expanding the header chrome.
- `buildVantageSummary()`: builds the short Vantage header summary and now omits the Library count when the Library dock is hidden, so the header does not advertise a temporarily removed surface.
- `buildDefaultSystemMessage()`: owns the first-run system introduction. It now introduces Vantage conversationally in first person, explaining that it helps manage context so work stays inspectable/reusable and can support Vantage inspection, whiteboard drafting, work-product publishing, Scenario Lab branching, and the current session mode, instead of duplicating that onboarding in the masthead.
- `renderExperimentStatus()`: keeps the masthead status user-facing. It now renders simple session status such as `Experiment mode`, `Connected`, or `Offline` instead of exposing implementation/provider labels like OpenAI, Lab, or Library in the top chrome.
- The default chat masthead now stays minimal: onboarding title/subtitle, experiment copy, and the experiment toggle are hidden from the header because the first system message carries that context.
- In the tighter production-whiteboard pass, `renderChatSurfaceCopy()` also shortens the whiteboard-mode masthead labels further so the narrow sidebar reads cleanly: the title becomes plain `Chat` and the sidebar action is labeled `Vantage` instead of the longer `Open Vantage`.
- That same mode-specific shell pass keeps the default composer at `Ask anything.` and swaps the placeholder in whiteboard mode to the shorter `Ask about this draft...`, so Vantage owns whiteboard routing instead of asking the user to mention it explicitly.
- `renderTurnPanel()`: central provenance-first answer-dock render path for the at-a-glance turn explanation, dedicated `Working Memory` summary, separate `Recall` and `Learned` groups, later `Memory Trace` / `Reasoning Path` detail, whiteboard update disclosure, and Scenario Lab separation.
- The Vantage answer dock no longer exposes the old turn-level controls for next-turn memory intent or immediate related-item opening. Those paths were removed from the visible UI and request payloads so Vantage reads as an inspection surface, not a control panel.
- `renderWorkingMemoryPanel()`: explains what was broadly in scope for generation without collapsing `Working Memory` into `Recall`, and surfaces compact facts such as always-in-scope request context, additional grounded sources, whiteboard scope hints, and pinned continuity.
- `renderLearnedCorrectionPanel()`: keeps the first-pass learned correction loop inside `What I learned`, with direct whiteboard / pinned-context actions and explicit non-direct guidance for wrong / temporary / forget requests that the backend does not support yet.
- `renderReasoningPathPanel()`: renders the staged inspection rail and stage drill-down panels from the `buildReasoningPathInspection()` view model, but keeps the whole Reasoning Path disclosure collapsed by default and later in the reading order until the user opens it.
- `renderMemoryTracePanel()`: renders recent-history contribution plus the created turn trace record without turning Memory Trace into a general library surface or blurring it into `Recall`.
- `renderScenarioLabPanel()`: renders the Scenario Lab review surface as a flagship comparison mode: question and recommendation centered in the hero, comparison-support cards next, the saved comparison hub as the durable anchor, branch details after that, and support rationale demoted later.
- `createScenarioTranscriptCard()`: adds a compact Scenario Lab result card directly in chat so the mode is legible before the user opens `Vantage`.
- `pushNotice()` / `renderNotices()`: now behave like a keyed replacement-style status surface in practice, showing only one transient top-right notice at a time instead of stacking multiple cards.
- `persistTurnSnapshot()` / `restoreTurnSnapshot()`: session-scoped client continuity.
- Read surfaces such as `addMessage()`, `createMemoryCard()`, and `renderMemoryInspector()` now use the shared `renderRichText()` helper from `math_render.mjs` so chat, ideas, notes, work products, and the whiteboard preview all parse LaTeX-style math, inline code, and fenced code the same way.

## Notable Behavior

- Uses the explicit surface enum from `surface_state.mjs` instead of interacting booleans.
- Keeps the whiteboard separate from `Vantage` inspection, with whiteboard-focused layout when drafting.
- Keeps whiteboard source editing and whiteboard rendering separate: the textarea remains editable source, while the preview below it is the read surface that renders LaTeX math, inline code, and fenced-code blocks.
- `renderChatSurfaceCopy()` now labels the whiteboard-focused chat rail as a sidebar so the shell reinforces one dominant drafting surface plus a secondary conversation rail rather than two equal work surfaces.
- The Pass 02A chat refinement keeps the chat-first copy short and editorial: default chat emphasizes provenance plus drafting surfaces in one sentence, while whiteboard-sidebar mode keeps only a reduced orienting line and the experiment note.
- Demotes experiment mode from a peer of `Whiteboard` / `Vantage` into a smaller session-level toggle next to the mode badge, so first-time users see fewer control-surface choices in the masthead.
- Reuses the same response-mode truth for compact chat evidence and the Vantage summary so the product difference stays visible without a full inspection click.
- Reframes the top Vantage dock as `This Turn` in presentation only while the inner turn panel stays provenance-first. The render path still keeps `Recall` distinct from broader `Working Memory` semantics and does not rename the underlying grounding model.
- Adds a Vantage-specific `Working Memory` explanation block ahead of `Recall`, so the first scan can answer what was in scope for generation even when that broader scope included whiteboard or recent-chat grounding instead of only recalled library items.
- Stabilizes the whiteboard masthead button back to `Whiteboard` for ordinary draft continuity instead of surfacing `Resume draft` as a headline control state.
- `addMessage()` now respects truth-layer evidence emphasis so lower-priority grounding chips render more quietly while stronger states like `Scenario Lab`, `Best Guess`, and durable `Learned` still read immediately in chat.
- Keeps the Scenario Lab compact transcript card parse-safe and boot-safe: malformed card logic here can block the entire module graph, so this renderer now uses the shared artifact-detail helper without introducing syntax-only branch-count logic.
- Centralizes turn grounding in the shared helper stack so the Vantage summary, turn panel, and Scenario Lab panel all consume the same normalized `response_mode` instead of re-deriving grounding semantics independently.
- Persists more than the visible draft: session snapshots are keyed by `{scope, experimentSessionId, workspaceId}` and also keep the active Reasoning Path stage, whether the Reasoning Path disclosure is expanded, turn-scoped Memory Trace cards, candidate trace notes, and other turn inspection state.
- Keeps `Recall` distinct from broader `Working Memory` framing in the inspection UI: `state.turnWorkingMemory` holds the surfaced recalled-item subset, while the `Working Memory` dock summary and Reasoning Path scope table explain the broader generation context.
- The `turnWorkingMemory` state name is still a compatibility alias for the recalled subset only. The UI now counterbalances that legacy name with an explicit `Working Memory` narrative layer and should keep that distinction documented until the compatibility seam is removed.
- Keeps deeper provenance secondary: the `Memory Trace` section now lives in the later `Look deeper` group, and `Reasoning Path` stays collapsed behind its own disclosure so the answer dock reads like guided explanation rather than a diagnostics wall.
- Carries `candidate_trace_notes` into the Considered context stage and `trace_notes` plus `memory_trace_record` into the Memory Trace section, so recent-history consideration, selected recent-history contribution, and the created turn trace record remain separate concepts in the UI.
- Uses a grouped explanation structure in the answer dock so `Working Memory`, `Recall`, `Learned`, `Memory Trace`, and `Reasoning Path` read as one ordered turn story instead of equal-weight sibling surfaces.
- Keeps the first visible provenance block shorter: the summary notice now uses the shared `buildTurnAtAGlanceSummary()` helper, and the support sections default to opening only the most immediately relevant group instead of expanding several at once.
- Reuses the existing card renderer inside Recall, Memory Trace, and Reasoning Path drill-down groups, but keeps `memory_trace` cards inspect-only and suppresses whiteboard-opening actions for stage drill-down cards.
- Shows item-level `Why recalled` lines only in Recall-oriented contexts, using the backend’s user-facing recall rationale when available and source-specific fallback copy otherwise.
- Makes learned items inspectable from canonical payload truth as well: learned cards, the inspector, and the turn-scoped correction panel consume `why_learned`, `correction_affordance`, `scope`, and `durability` when present, expose `Saved because`, and keep correction truthful by preferring whiteboard revision plus pinned-next-turn continuity over fake delete or scope-mutation actions.
- Tightened the learned-item contract so the visible labels match the actual behavior: pinning now reads as `Pin as context` / `Unpin context`, and the unsupported wrong / temporary / forget controls are presented as guidance selectors rather than mutation buttons.
- Keeps learned correction state scoped to the active turn item instead of pretending durable mutation happened: the panel can switch between direct correction, mark-wrong guidance, make-temporary guidance, and forget guidance, but unsupported paths stay explicitly labeled `Not direct yet`.
- Makes the learned-item correction loop more explicit without inventing unsupported storage mutation: learned cards now surface a compact correction-path hint, offer `Pin as context` directly from the card, and the inspector adds a dedicated correction-options block that explains what can be done now versus what still requires a future backend mutation path.
- Carries `scope` and `durability` through normalized saved-item records so the learned-item UI can distinguish temporary experiment saves from durable library saves without inferring that state from labels alone.
- Softens the remaining whiteboard lifecycle copy so draft transitions read like collaborative writing help instead of state reconciliation, including gentler phrasing for fresh drafts, continued drafts, and accepted whiteboard drafts.
- Keeps raw vetting rationale out of the visible `This Turn` notice. When nothing is selected into Recall, the notice stays product-level instead of surfacing backend/debug wording.
- Uses the boot-only workspace reconciliation helper from `workspace_state.mjs` so `restoreTurnSnapshot()` and `loadWorkspace()` agree on when a restored draft should survive, when stale Vantage inspection should collapse, and when a fresh backend workspace should replace the restored one, including blank-to-saved or saved-to-blank workspace-anchor changes.
- Applies the same explicit continuity rule to the whiteboard-owned latest-artifact cue: the cue only survives passive reload when the incoming workspace still matches the current whiteboard’s id, scope, and normalized content, so same-id replacements do not advertise stale artifact continuation while harmless title or whitespace normalization does not wipe the cue.
- Uses the normalized interpretation payload directly in the Route stage, including requested whiteboard mode, resolved whiteboard mode, humanized route confidence, and kept-in-scope continuity labels.
- Softens the remaining inspection language so the Reasoning Path reads like guided provenance instead of telemetry, including `Considered context`, `used for recall`, and `Kept in scope`.
- Keeps Scenario Lab confidence calmer in the main surface by using qualitative route language such as `Clear fit`, `Strong fit`, `Possible fit`, and `Tentative fit` instead of precise percentages or awkward adjective-only summaries.
- Trusts the canonical normalized `recallCount` for grounding summaries instead of relying only on the currently visible recalled-item array length.
- Uses top-level turn Recall, Learned, and response-mode state when describing Scenario Lab grounding, rather than assuming those fields live inside the Scenario Lab payload itself.
- Frames Scenario Lab as a dedicated comparison-first reasoning surface, separate from the Working Memory dock, and surfaces a compact open-the-lab card in chat for successful Scenario Lab turns.
- Orders the Scenario Lab panel around a clearer product story: comparison question and recommendation first, durable comparison hub second, reopenable branches next, and route/grounding support later as secondary inspection context.
- Moves the strongest branch affordances back to the comparison hub layer so the durable hub remains the main reopen/inspect anchor, while the separate branch section acts as the richer read surface for comparing branch details.
- Uses the comparison hub’s indexed branch roster as the durable revisit hub in the Scenario Lab panel and the compact chat card, so reopen/inspect actions stay anchored to the saved comparison instead of only the branch workspaces.
- Keeps the main Scenario Lab panel intentionally non-duplicative: the comparison question and recommendation lead the panel, the comparison hub card owns the compact branch roster and reopen actions, and the separate branch-detail section preserves the richer per-branch reasoning cards.
- Demotes Scenario Lab support rationale further by dropping provenance-style mini counters from the support cards, so the slice stays closer to a coherent reasoning mode than to ordinary Vantage provenance UI.
- Softens branch-level confidence in Scenario Lab as well, so branch cards favor calmer labels like `Balanced case` instead of surfacing raw confidence strings where the source text can sound overly exact.
- Normalizes comparison-hub rendering through the same artifact-detail helper the transcript card uses, so the compact chat card and the full Scenario Lab panel agree on branch counts, labels, and inspect actions.
- Reuses the shared `normalizeComparisonBranchIndex()` helper from `turn_payloads.mjs` when renderer-side Scenario Lab fallbacks need a branch roster, so the payload-normalization and render paths do not drift into separate branch-index logic or hit missing helper errors.
- Keeps branch-detail inspection local to Scenario Lab: the comparison hub can focus a matching branch card in place, and the focused card is highlighted and scrolled into view instead of opening a second inspector surface.
- Persists the currently inspected Scenario Lab branch inside the turn snapshot so the focused branch detail state survives ordinary rerenders and same-turn reopen behavior.
- Treats unsaved dirty whiteboards as authoritative during passive refresh so the backend's last saved workspace does not erase a local draft.
- Stores both workspace-scoped and scope-scoped turn snapshots so temporary unsaved drafts can survive across refreshes and workspace-id changes.
- Treats whiteboard offers and draft proposals as pending decisions until the user applies or accepts them.
- Treats the latest saved artifact as whiteboard-owned lifecycle state rather than turn-owned state: the whiteboard surface can show a compact latest durable work product cue with inspect/reopen actions without turning the artifact into a new drafting surface.
- The Pass 02B whiteboard refinement keeps that lifecycle cue quieter and more page-integrated while preserving explicit lifecycle wording from the artifact model instead of flattening saved snapshots and promoted artifacts into the same label.
- Treats artifacts as a first-class Library-inspector variant instead of inventing a new artifact surface: the inspector now distinguishes durable work products and comparison hubs from ordinary memories while keeping review read-only and continuation explicit through reopen actions.
- Normalizes the new thin lineage payload fields (`derived_from_id`, `revision_parent_id`, `lineage_kind`) alongside raw `comes_from`, plus explicit artifact presentation fields (`artifact_origin`, `artifact_lifecycle`), then renders a lightweight lifecycle cue such as `saved whiteboard`, `promoted artifact`, or `comparison hub` without implying that every provenance edge is a formal revision chain.
- Applies a narrower restore policy for scope-scoped fallback snapshots: they keep explicit pinned context and draft continuity, but they clear stale inspected-record detail state, collapse any stale `Vantage` reopen back to the return surface, and still restore the exact `workspaceContextScope` so the reopened turn remains truthful about whether whiteboard content was in scope.
- Clears Scenario Lab branch-detail focus during scope-scoped fallback restores for the same reason it clears other inspected-record detail state: fallback continuity can keep the turn, but it should not silently revive a stale branch-inspection substate.
- When the backend returns an auto-saved artifact snapshot for a whiteboard save, the client upserts it into the saved-note catalog and surfaces a dedicated notice without changing the whiteboard lifecycle into a promoted artifact.
- Artifact cards and the inspector now prefer explicit lifecycle wording over brittle provenance guesses: saved whiteboard snapshots read as whiteboard saves, promoted artifacts read as promotions, and Scenario Lab artifacts read as comparison hubs while still using the same inspect/reopen affordances.
- Keeps the whiteboard latest-work-product cue action-light and visually secondary: `Inspect work product` reuses the existing `Vantage` library inspector, while `Reopen in whiteboard` reuses the existing guarded saved-item reopen path instead of inventing a parallel artifact route.
- Standardizes action language around the current contract: `Inspect` means read-only review in `Vantage`, while work products use `Reopen in whiteboard` to continue from a durable saved version rather than implying continuity from inspection alone. Scenario Lab comparison hubs now also follow that inspect-first ordering.
- Recognizes the generic `open_saved_item_into_workspace` graph action as the canonical reopen label while still accepting the older `open_concept_into_workspace` value during the compatibility window.
- Routes reveal/hide whiteboard transitions through the shared surface helpers rather than ad hoc `{ current, returnSurface }` assignments.
- Forces pending whiteboard context explicitly for the dedicated accept-button flow, so `/api/chat/whiteboard/accept` does not rely on the ordinary chat carry matcher for empty-message acceptance.
- Uses canonical `record_id` when opening saved items into the whiteboard, while keeping any remaining id-compatibility fallback centralized in the DTO helpers instead of spread across DOM code.
- Resolves a small but important continuity seam between Recall and Whiteboard: if the user immediately asks to pull `that` into the whiteboard after the last turn surfaced one clear durable item, the client reuses the existing reopen path instead of sending a brittle fresh chat request that might stay anchored to the wrong whiteboard draft.
- Uses one shared renderer seam for math-capable text surfaces instead of sprinkling KaTeX calls throughout the UI. That keeps chat, concept cards, the library inspector, and the whiteboard preview aligned, and keeps the mutable source surfaces free from HTML mutation logic.
- Avoids a noisy top-right rail by no longer stacking payload-derived graph notices. The rail is now reserved for the single most important transient feedback, while whiteboard and provenance state stay inline on their own surfaces.
